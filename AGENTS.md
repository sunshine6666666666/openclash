# Cudy TR3000 硬件性能与部署红线

本规则记录了当前设备的真实硬件参数与性能边界，用于指导软件安装、插件配置及系统优化建议。

## 1. 核心硬件规格 (Hardware Specs)
- **型号**: Cudy TR3000 (AX3000)
- **架构**: ARMv8 Processor rev 4 (aarch64, 64-bit)
- **CPU**: 双核 MediaTek Filogic 820
- **内存 (RAM)**: 512MB (可用约 200MB+)
- **闪存 (Flash)**: 256MB (Overlay 剩余空间约 160MB+)
- **系统**: ImmortalWrt 24.10.4 (Kernel 6.6.110)

## 2. 软件基线 (Software Baseline)
- **OpenClash App**: v0.47.055
- **DNS组件**: dnsmasq-full (v2.90-r5) - *已确认可完美支持 DNS 劫持与过滤*
- **内核架构**: linux-arm64 (v8)
- **关键依赖**: kmod-tun (v6.6.110-r1), curl (v8.12.1-r1), bash (v5.2.37-r1)

## 3. 运行健康基准 (Health Baseline - 2026-02-12)
- **典型工作温度**: 58°C - 62°C (空闲/轻载)
- **进程监控**: clash/mihomo 主进程 + watchdog 守护进程
- **网络转发**: 运行在 Fake-IP 模式 (Redir-Host), 核心转发通过 eth1
- **系统负载**: Load Average 维持在 < 0.1 (性能冗余极大)

## 4. 性能边界 (Performance Boundaries)
- **✅ 推荐部署**: OpenClash (Meta), AdGuard Home, DDNS, WOL。
- **⚠️ 谨慎部署**: 极轻量 Docker (内存 <100MB), 频繁读写应外挂硬盘。
- **❌ 禁止部署**: `opencode`, `openclaw` 等重型 AI Agent, Chromium 等内存巨兽。

## 5. 基线一致性约束 (Baseline Alignment Constraint)
- **100% 对齐要求**: 任何在机器上安装的软件包或重大配置更改，**必须立即同步更新**到本文件的 `2. 软件基线`。
- **记录规范**: 必须包含确切版本号（如 `v2.90-r5`），严禁“装了但不记”。

## 6. 决策原则 (Decision Principles)
1. **基线优先**: 所有运维决策必须基于本文件最新基线。
2. **稳定性优先**: 优先保证路由器网络转发本职工作。
3. **内外分离**: 重型计算（AI、大型数据库）必须放在 Mac 宿主机，路由器仅作为网关。
4. **空间预警**: 保持 Overlay 剩余空间不少于 50MB。

## 7. OpenClash UI 时效约束与 5 分钟流程（强制）
- **时效前提**: 新机场订阅链接常见仅 5 分钟有效，过期后会出现 `401/403/timeout`。
- **网络前提**: 更新订阅时必须优先保证“国内直连可用”；若代理链路不稳，先停 OpenClash 再更新。
- **界面前提**: 在 `服务 -> OpenClash -> 配置订阅` 页面操作；只有“保存配置 + 更新”都完成才算生效。

### 7.1 UI 字段标准写法（避免填错）
- **配置文件名**: `Bolster`（可自定义，但建议固定命名便于识别）
- **订阅地址**: 机场最新复制的原始链接（不要用过期链接）
- **User-Agent**: `clash.meta`
- **在线订阅转换**: 不确定时先关闭转换模板，优先验证原始链接可拉取
- **启用**: 新订阅临时启用，旧失败订阅先禁用，避免并发更新干扰

### 7.2 五分钟一次性完成 SOP（团队执行标准）
1. **T+00:00** 先点机场“复制订阅链接”拿到最新链接（不要提前复制）。
2. **T+00:30** 进 OpenClash 的 `配置订阅`，编辑 `Bolster` 粘贴新链接，`User-Agent=clash.meta`，点“保存配置”。
3. **T+01:30** 回列表页仅点 `Bolster` 这一行“更新”，不要同时更新其他订阅。
4. **T+03:00** 到顶部“当前配置文件”下拉，确认是否出现并可切换到新配置文件。
5. **T+04:00** 回 `运行状态` 观察访问检查，至少确认国内站可用；再验证国外站。
6. **T+05:00** 若失败，立即停用该新订阅并回退到当前可用配置，不做多轮盲试。

### 7.3 失败判定与动作
- **`401`**: 鉴权或配额限制（token/IP/来源限制），需让客服重置或提供 OpenClash 专用链接。
- **`403`**: 区域/网络策略拒绝，优先更换网络环境或让客服放行。
- **`Could not resolve host`**: 本机 DNS 链路异常，先恢复 DNS 再重试。

## 8. 当前可用策略（实战结论，默认执行）
- **收工判定**: 只要 `国内OK + 国外OK` 同时成立，立即停止改动，进入稳定使用。
- **禁止过度优化**: 在已可用状态下，不再反复切模式、切规则组、切节点，避免“越改越坏”。
- **本订阅现实约束**: 当前机场配置更偏“全局模式”可用；规则模式可留待后续优化，不作为当前必做项。
- **最小改动原则**: 仅维护 1 个主节点 + 1 个备用节点；出现连续故障再切换，不做频繁人工切换。

### 8.1 日常运维三条线（固定流程）
1. **先看结果**: 先跑连通性检查（国内/国外）再决定是否改配置。
2. **只改一处**: 每次最多改 1 个开关或 1 个节点，改后立即复测。
3. **可用即冻结**: 一旦恢复双 OK，冻结配置，不做额外“优化型”操作。

### 8.2 一分钟应急口令（可直接执行）
```bash
curl -sS --max-time 5 https://www.baidu.com >/dev/null && echo 国内OK || echo 国内FAIL; curl -sS --max-time 8 https://api.ipify.org >/dev/null && echo 国外OK || echo 国外FAIL
```

判定动作：
- **国内OK + 国外OK**: 不动
- **国内OK + 国外FAIL（连续2次）**: 切备用节点
- **国内FAIL**: 先停 OpenClash，恢复国内直连

## 9. 机场节点访问限制与策略 (Airport Node Restrictions & Policies)
根据机场最新公告，节点访问遵循以下端口与区域过滤规则：

### 9.1 通用基础规则
- **默认放行**: 80, 443 (Web), 1024-9999 (主动端口)。
- **默认屏蔽**: 25 (SMTP 发信)。
- **特殊限制**: 针对限制访问端口，**仅放行域名请求**（如 `github.com:22`）及白名单 IP，**拦截纯 IP 请求**（如 `1.1.1.1:22`）。

### 9.2 区域屏蔽等级 (Regional Filtering Levels)
- **等级 1 (香港、日本、新加坡、台湾)**:
    - **放行**: 10001-65535 (被动端口), 110, 993, 995, 143 (邮件收信)。
- **等级 2 (除美国、韩国、欧洲外的其他节点)**:
    - **放行**: 110, 993, 995, 143 (邮件收信), 10001-65535 (被动端口)。
    - **屏蔽**: 465, 587 (邮件发信), 22 (SSH)。
- **等级 3 (美国、日本绝大部分、韩国、欧洲核心区)**:
    - **屏蔽**: 465, 587 (邮件发信), 22 (SSH), 110, 993, 995, 143 (邮件收信), 10001-65535 (被动端口)。
